# lid — design

`lid` is the lid on the ghost jar: an agent-containment CLI that runs `claude`
(or any agent) inside a matchlock microVM with a network allowlist and
host-side credential injection. Secrets never enter the VM; the local network
is unreachable unless explicitly allowed.

## Goals

1. **VM isolation** — the agent runs inside a matchlock microVM
   (Virtualization.framework on macOS, Firecracker on Linux).
2. **Fast startup** — `lid` in a project directory boots and attaches in
   roughly the time matchlock takes to boot (<1s warm); no more friction than
   running `claude` directly.
3. **Network isolation** — deny-by-default. No network config means *no NIC at
   all*. An allowlist means MITM interception with private IP ranges blocked
   unless the config explicitly opts out.

## Architecture

```
lid.star (project) + ~/.config/lid/lid.star (global)
        │  starlark evaluation (go.starlark.net), pure & hermetic
        ▼
internal/config: File{Profiles: []*Profile}   ← the only "smart" layer; heavily
        │                                        property-tested (pgregory.net/rapid)
        ▼
internal/runner: resolve secret sources on the host (env / command / gh auth token),
        │        translate Profile → matchlock sdk.SandboxBuilder, launch,
        │        guest bootstrap (git credential helper), interactive TTY exec
        ▼
matchlock `rpc` subprocess (JSON-RPC over stdio) → microVM
```

- **cmd/lid** — CLI (stdlib `flag`, hand-rolled subcommands): `run`, `shell`,
  `ls`, `check`.
- **internal/config** — starlark → validated `Profile`. Pure: no I/O, no
  clock, no env access during evaluation. All security-relevant normalization
  (allowlist union, private-IP defaults, no-network defaults) happens here so
  it is testable.
- **internal/runner** — everything effectful. Kept as thin as possible.

## Key decisions

- **Deny-by-default inversion.** Matchlock treats an *empty* allowlist as
  "allow all". lid inverts this: omitted network ⇒ `WithNoNetwork()`;
  a network block requires either a non-empty `allow` list or an explicit
  `allow_all=True`. lid never emits "interception with empty allowlist"
  by accident.
- **Private IPs blocked always-explicitly.** Compiled profiles always set
  block-private-IPs explicitly (never rely on matchlock defaults);
  `allow_private=True` is the only way out.
- **Secret sources resolve at launch, not at config time.** Config carries
  *where* a secret comes from (`from_env`, `from_cmd`, github helper), so
  config evaluation stays pure and secret values can't leak into config state.
- **Secret hosts are unioned into the allowlist.** A secret you configured is
  a statement of intent that its hosts be reachable.
- **GitHub credential injection** uses matchlock's MITM basic-auth-aware
  placeholder replacement (pkg/policy handles base64-encoded
  `Authorization: Basic` values). The runner:
  1. resolves a token on the host (`$GITHUB_TOKEN` / `$GH_TOKEN`, else
     `gh auth token`),
  2. registers it via `AddSecretWithPlaceholder` (placeholder generated by
     lid so it can be referenced in guest env, e.g. `GH_TOKEN`),
  3. bootstraps the guest with a git credential helper that answers
     `username=x-access-token`, `password=$GITHUB_TOKEN` (the placeholder).
  The real token never enters the VM.
- **Config is starlark** via go.starlark.net with default safety posture (no
  `while`, no recursion, no I/O builtins) plus an execution-step cap, so a
  hostile config can't loop forever or touch the host.
- **cwd is mounted read-write at `/workspace` by default** — the point is for
  the agent's edits to land in your project. `mount_cwd="ro"|"off"` for stricter
  runs.
- **matchlock dependency** is the local checkout at `~/co/matchlock` via a
  `replace` directive (SDK is unversioned upstream; this repo is
  machine-specific dotfiles anyway).

## Verification strategy

Tests for `internal/config` were written by a separate agent from `SPEC.md`
*before* the implementation existed, pinned against a stub API, and the
implementation agent was forbidden from editing test files (enforced by
diffing test files against the pre-implementation jj commit). Property tests
(rapid) generate random config values, render them to starlark source,
evaluate, and check the invariants in SPEC.md §Invariants — notably that no
generated config can ever produce a plan with an open network or unblocked
private IPs unless it explicitly asked for it, and that secret values never
appear outside secret specs.

## Future / stretch

- Image building (bake `claude` + toolchains into an OCI image; `lid bake`).
  For now: any image with node/npm works, e.g. `docker.io/library/node:22`,
  with `npm install -g @anthropic-ai/claude-code` in setup, or a prebuilt
  image ref in config.
- Overlay (snapshot) mounts for cwd once matchlock's overlay VFS type is the
  right fit.
- Per-run allowlist mutation (`lid allow <host>` against a running VM).
